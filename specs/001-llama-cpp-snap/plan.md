---

## Summary

Build a strictly confined Ubuntu snap for `llama.cpp` (`llama-server`) that:

1. Builds from upstream llama.cpp source with the Vulkan GPU backend.
2. Defaults to zero on-disk inference data (no KV cache, no prompt cache on disk).
3. Provides an `se-llama.server` command with `presets.ini`-based parameter management.
4. Provides an `se-llama.update-models` command for syncing presets into an OpenCode config.
5. Passes `snap lint` with strict confinement and a minimal interface set.

GPU support priority: Vulkan (v1). ROCm and CUDA accommodated architecturally for v2+.

---

## Technical Context

**Language/Version**: C++17 (llama.cpp upstream); shell (wrapper scripts); Python 3 for the OpenCode config sync helper.

**Primary Dependencies**:
- llama.cpp (upstream `https://github.com/ggml-org/llama.cpp`, tip of main or latest stable tag)
- CMake в‰Ґ 3.14 (build)
- Vulkan SDK: `libvulkan-dev`, `glslc`, `spirv-headers` (build)
- `libvulkan1` (runtime, staged from Ubuntu 24.04)
- snapcraft (packaging)

**Storage**:
- `$SNAP_USER_COMMON/models/` вЂ” user GGUF model files (persists across snap refresh)
- `$SNAP_USER_COMMON/config/presets.ini` вЂ” user config (persists)
- `$SNAP_USER_DATA/logs/` вЂ” server logs (per-revision)
- No inference data written to disk in default configuration

**Testing**:
- Snap build validation: `snap lint`
- Integration: start server with test model, poll `/health`, verify zero cache files post-stop
- Confinement test: attempt out-of-bounds file access, verify AppArmor denial
- Preset test: start with named preset, verify parameters applied via `/props` endpoint

**Target Platform**: Ubuntu 24.04 LTS, amd64. snapd в‰Ґ 2.60.

**Project Type**: snap package (CLI tool + HTTP server)

**Performance Goals**:
- Server ready (passes `/health`) within 30 seconds of start on в‰Ґ8 GB RAM machine
- Build completes in в‰¤30 minutes on a modern workstation
- No measurable inference throughput regression vs. bare llama-server binary

**Constraints**:
- Strict confinement MUST pass `snap lint` with no errors
- No GPU vendor libraries bundled in snap (must use host ICDs via `opengl` interface)
- No persistent inference data in default preset
- CUDA/ROCm builds require separate build infrastructure (out of scope for v1, but `snapcraft.yaml` structure must accommodate future build-flag variants)

**Scale/Scope**: Single-user local inference. Not designed for multi-user server deployment in v1.

## Phase 0: Research Summary

All NEEDS CLARIFICATION items resolved. See `research.md` for full findings.

Key decisions:
- **Build**: CMake + Ninja, `-DGGML_VULKAN=1 -DBUILD_SHARED_LIBS=OFF -DCMAKE_INSTALL_PREFIX=/usr`
- **GPU interface**: `opengl` snapd interface (no dedicated `vulkan` interface exists)
- **Model access**: snap-owned `$SNAP_USER_COMMON/models/` directory; no extra interface is required for server access to snap data
- **Cache suppression**: omit `--slot-save-path`; default `presets.ini` uses `no-cache-prompt = true` and `cache-ram = 0`
- **Config**: `presets.ini` passed natively via `--models-preset`; llama-server parses it directly (router mode). Wrapper lazily seeds the file on first run вЂ” no custom INI parsing and no install hook required.
- **Router mode**: Server started without `--model`; clients select model per-request via `"model"` field in API call matching a `presets.ini` section name/alias.
- **Wrapper**: Minimal вЂ” prepend `--models-preset` and `--models-dir` flags, seed config, exec llama-server.
- **Multi-user purge caveat**: documented in quickstart.md; requires manual action for other users

## Complexity Tracking

| Complexity | Why Needed | Simpler Alternative Rejected Because |
|---|---|---|
| Wrapper script (seeds config, prepends --models-preset) | Snap needs a first-run config seed mechanism and a stable entry point | Passing all flags via CLI args each time is error-prone and not auditable for SE use |
| Two snap apps (`server` + `update-models`) | Different plug sets per principle III | Single app would require config sync to inherit server GPU and bind permissions unnecessarily |
| Vulkan build deps (`glslc`, `spirv-headers`) | llama.cpp Vulkan backend requires shader compilation at snap build time | CPU-only build rejected; GPU acceleration is a stated requirement |
